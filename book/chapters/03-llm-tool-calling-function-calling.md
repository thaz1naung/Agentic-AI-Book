# 03-llm-tool-calling-function-calling

### စကားလုံးမှသည် လုပ်ဆောင်ချက်ဆီသို့

ကျွန်ုပ်တို့သည် ယခင်အခန်းများတွင် LLM ဟူသော "ဦးနှောက်" အကြောင်းနှင့် ၎င်းကို ထိန်းကျောင်းမည့် "လုပ်ငန်းစဉ်" အကြောင်းများကို ဆွေးနွေးခဲ့ကြသည်။ သို့သော် ဦးနှောက်သည် မည်မျှပင် တော်စေကာမူ၊ ပြင်ပလောကနှင့် ဆက်သွယ်နိုင်သော "လက်" များ မရှိပါက ၎င်းသည် အခန်းပိတ်၍ စာရေးနေသော ပညာရှိတစ်ဦးနှင့်သာ တူပေလိမ့်မည်။ ထိုပညာရှိသည် သင့်အား "ဟင်းချက်နည်း" ကို ပြောပြနိုင်သော်လည်း၊ သူကိုယ်တိုင် မီးဖိုချောင်ထဲသို့ သွား၍ ဟင်းချက်ပေးနိုင်မည် မဟုတ်ချေ။ Agentic AI လောကတွင် ထို "လက်" များကို ကျွန်ုပ်တို့က **Tool Calling** သို့မဟုတ် **Function Calling** ဟု ခေါ်ကြသည်။

ဤနေရာတွင် လူအများ မှားယွင်းနေသော အချက်တစ်ခု ရှိသည်။ ၎င်းမှာ LLM သည် Software များကို တိုက်ရိုက် ကိုင်တွယ် မောင်းနှင်နေသည်ဟု ထင်မှတ်နေခြင်း ဖြစ်သည်။ အမှန်စင်စစ် LLM သည် မည်သည့် Software ကိုမျှ တိုက်ရိုက် မနှိပ်နိုင်၊ မမောင်းနိုင်ချေ။ ၎င်းသည် "စာရေးတော်" သာ ဖြစ်သည့်အတွက် ၎င်းလုပ်နိုင်သည်မှာ "စာရေးခြင်း" သာ ဖြစ်သည်။ 

သို့သော် ထိုစာသည် သာမန်စာမဟုတ်ဘဲ၊ စက်ပစ္စည်းများ နားလည်သော **JSON (JavaScript Object Notation)** ဟူသော နည်းပညာဘာသာစကားဖြင့် ရေးသားထားသော "အမိန့်စာ" ဖြစ်သည်။ ကျွန်ုပ်တို့၏ တာဝန်မှာ ထိုအမိန့်စာကို ဖတ်ရှုပြီး လက်တွေ့ အကောင်အထည်ဖော်ပေးမည့် "လက်ထောက်" (Harness/Runtime) တစ်ခု တည်ဆောက်ပေးရန် ဖြစ်သည်။

### နည်းပညာဆိုင်ရာ ပဋိညာဉ် (The Technical Contract)

Tool Calling ၏ အလုပ်လုပ်ပုံကို နားလည်ရန် "စာချုပ်" တစ်ခုကို မြင်ယောင်ကြည့်ပါ။ ကျွန်ုပ်တို့သည် LLM အား "မင်းမှာ ဒီလက်နက်ကိရိယာတွေ ရှိတယ်" ဟု ကြိုတင် အသိပေးရသည်။ ဤသည်ကို **Tool Definition** ဟု ခေါ်သည်။ ဥပမာအားဖြင့်၊ ကျွန်ုပ်တို့သည် "ဂဏန်းပေါင်းစက်" ဟူသော Tool တစ်ခုကို ပေးအပ်လိုလျှင် LLM အား အောက်ပါအတိုင်း ပြောပြရမည် -

> "မင်းမှာ `add_numbers` ဆိုတဲ့ Tool ရှိတယ်။ အဲဒီ Tool ကို သုံးချင်ရင် မင်းက `a` နဲ့ `b` ဆိုတဲ့ ကိန်းဂဏန်း နှစ်ခုကို ပေးရမယ်။ အဲဒါဆိုရင် ငါက အဲဒီကိန်းဂဏန်းတွေကို ပေါင်းပေးမယ်။"

ဤသည်မှာ LLM နှင့် ကျွန်ုပ်တို့ ကြားရှိ "ပဋိညာဉ်" ပင် ဖြစ်သည်။ LLM က "၅ နဲ့ ၁၀ ကို ပေါင်းပေးပါ" ဟူသော အမိန့်ကို ရရှိသောအခါ၊ ၎င်းသည် သာမန်စာသားဖြင့် ပြန်မဖြေတော့ဘဲ၊ ထိုပဋိညာဉ်အတိုင်း `{"tool": "add_numbers", "arguments": {"a": 5, "b": 10}}` ဟူသော JSON စာသားကို ထုတ်ပေးသည်။ ကျွန်ုပ်တို့၏ Software က ထို JSON ကို ဖတ်ပြီး၊ အမှန်တကယ် ပေါင်းစပ်ကာ ရလဒ် `15` ကို LLM ထံ ပြန်လည် ပေးပို့ရသည်။ ဤသို့ဖြင့် LLM သည် ၎င်းကိုယ်တိုင် သင်္ချာမတွက်တတ်သော်လည်း၊ ဂဏန်းပေါင်းစက်ကို အသုံးပြု၍ မှန်ကန်သော အဖြေကို ပေးနိုင်လာခြင်း ဖြစ်သည်။

ဤနေရာတွင် သတိပြုရန်မှာ LLM သည် "လူမိုက်" နှင့် တူသည်ဟု ကျွန်ုပ် ခဏခဏ ပြောခဲ့ဖူးသည်။ အကယ်၍ သင်၏ ပဋိညာဉ် (Definition) သည် မရှင်းလင်းပါက LLM သည် ယောင်မှားပြီး မှားယွင်းသော အမိန့်စာများကို ထုတ်ပေးပေလိမ့်မည်။ ၎င်းကို **Schema Violation** ဟု ခေါ်သည်။ ထို့ကြောင့် အင်ဂျင်နီယာတစ်ယောက်အနေဖြင့် Tool များကို အဓိပ္ပာယ်ဖွင့်ဆိုရာတွင် အလွန် တိကျရမည်။ "Description" ကဏ္ဍတွင် "ဤ Tool သည် ဂဏန်းပေါင်းရန် ဖြစ်သည်" ဟု ရိုးရိုးလေး ရေးမည့်အစား "ဤ Tool သည် ကိန်းပြည့် (Integer) နှစ်ခုကိုသာ လက်ခံပြီး ၎င်းတို့၏ ပေါင်းလဒ်ကို ပြန်ပေးသည်" ဟု တိတိကျကျ ရေးသားပေးရမည်။

### အသိဉာဏ်နှင့် လက်နက်တို့၏ အားပြိုင်မှု

LLM များသည် ၎င်းတို့ကိုယ်တိုင် သင်္ချာတွက်ခြင်း သို့မဟုတ် ကုဒ်ရေးခြင်းထက်၊ Tool များကို အသုံးပြုခြင်းက ပိုမို ထိရောက်သည်ကို ကျွန်ုပ်တို့ တွေ့ရှိရသည်။ ဤသည်မှာ လူသားတို့၏ တိုးတက်မှု သမိုင်းနှင့်လည်း တူညီသည်။ 

လူသားသည် မိမိ၏ လက်ဗလာဖြင့် ကျောက်တုံးကြီးကို မမနိုင်သော်လည်း၊ "မောင်းတံ" ဟူသော Tool ကို သုံး၍ မနိုင်ခဲ့ကြသည်။ Agentic AI သည်လည်း ထိုသို့ပင်။ ၎င်း၏ အတွင်းပိုင်း "ခန့်မှန်းချက်" (Inference) သည် အမြဲတမ်း မမှန်နိုင်သော်လည်း၊ တိကျသော Tool များကို အသုံးပြုခြင်းဖြင့် ၎င်း၏ အားနည်းချက်များကို ဖုံးကွယ်နိုင်သည်။

သို့သော် ဤနေရာတွင် အန္တရာယ်တစ်ခု ရှိသည်။ 

၎င်းမှာ **Tool Over-reliance** (Tool ကို အလွန်အကျွံ အားကိုးခြင်း) ဖြစ်သည်။ တစ်ခါတစ်ရံတွင် LLM သည် ရိုးရိုးလေး ဖြေရမည့် မေးခွန်းကိုပင် Tool များ အဆင့်ဆင့် ခေါ်ယူပြီး ရှုပ်ထွေးအောင် လုပ်တတ်သည်။ 

ဥပမာ - "မင်္ဂလာပါ" ဟု နှုတ်ဆက်သည်ကိုပင် "ပြန်လည်နှုတ်ဆက်ခြင်းဆောင်ရွက်သော API"  ရှိ မရှိ လိုက်ရှာနေတတ်သည်။ ဤသည်မှာ စွမ်းအင်နှင့် အချိန်ကို ဖြုန်းတီးခြင်းပင် ဖြစ်သည်။ ထို့ကြောင့် ကျွန်ုပ်တို့သည် Agent အား မည်သည့်အချိန်တွင် Tool ကို သုံးရမည်၊ မည်သည့်အချိန်တွင် မိမိ၏ ဉာဏ်ကို သုံးရမည် ဆိုသည်ကိုပါ သင်ကြားပေးရမည် (System Prompting မှတစ်ဆင့်)။

နောက်ထပ် အရေးကြီးသော အချက်မှာ **Multi-tool Orchestration** ဖြစ်သည်။ 

ရှုပ်ထွေးသော အလုပ်တစ်ခုတွင် Agent သည် Tool တစ်ခုတည်းဖြင့် မလုံလောက်ချေ။ 

ဥပမာ - "ယနေ့ သတင်းများကို ရှာပြီး အနှစ်ချုပ်ကို အီးမေးလ်ပို့ပါ" ဟု ခိုင်းလျှင် Agent သည် ပထမဦးစွာ "Search Tool" ကို သုံးရမည်၊ ရရှိလာသော အချက်အလက်များကို ဖတ်ရှုရမည်၊ ထို့နောက် "Summarization" ကို (မိမိဉာဏ် LLM inference ဖြင့်) လုပ်ရမည်၊ နောက်ဆုံးတွင် "Email Tool" ကို သုံးရမည်။ ဤအဆင့်ဆင့်သော လုပ်ငန်းစဉ်ကို Agent က မိမိဘာသာ စီစဉ်နိုင်ခြင်းသည် Agentic AI ၏ အစွမ်းထက်ဆုံးသော အချက်ပင် ဖြစ်သည်။

### လုံခြုံရေးနှင့် အာဏာစက်

Tool Calling ကို ခွင့်ပြုလိုက်ခြင်းသည် Agent အား သင်၏ Software စနစ်အတွင်းသို့ ဝင်ရောက်ခွင့် ပေးလိုက်ခြင်း ဖြစ်သည်။ ဤသည်မှာ "သူစိမ်းတစ်ယောက်ကို သင့်အိမ်ထဲသို့ ဖိတ်ခေါ်ပြီး အိမ်အကူ လုပ်ခိုင်းခြင်း" နှင့် တူသည်။ အကယ်၍ ထိုသူစိမ်းသည် သင်ခိုင်းသည့်အတိုင်း မလုပ်ဘဲ၊ သင့်အိမ်ရှိ ရွှေငွေများကို ခိုးယူသွားပါက မည်သို့လုပ်မည်နည်း။

Software Engineering ရှုထောင့်မှ ကြည့်လျှင် ဤသည်မှာ **Security Vulnerability** ဖြစ်သည်။ Agent သည် Tool Definition ထဲတွင် မပါသော အမိန့်များကို ထုတ်ပေးရန် ကြိုးစားနိုင်သည် (သို့မဟုတ် အပြင်လူ၏ လှည့်စားမှုကို ခံရနိုင်သည်)။ 

ဥပမာ - `delete_user` ဟူသော Tool ကို သင်က ပေးမထားသော်လည်း၊ Agent က `{"tool": "delete_user", "arguments": {"id": "admin"}}` ဟု ယောင်မှား၍သော် လည်းကောင်း၊ တစ်ဦးတစ်ယောက်၏ လှည့်စားမှုကြောင့်သော် လည်းကောင်း ထုတ်ပေးလာနိုင်သည်။ 

အကယ်၍ သင်၏ Runtime က ဤအမိန့်ကို မစစ်ဆေးဘဲ ထို tool calling ကို အကောင်အထည်ဖော်လိုက်ပါက သင်၏ စနစ်ကြီးတစ်ခုလုံး ပျက်စီးသွားပေလိမ့်မည်။

ထို့ကြောင့် ကျွန်ုပ်တို့သည် **Sandboxing** နှင့် **Validation** စနစ်များကို မဖြစ်မနေ ထည့်သွင်းရမည်။ Agent က ပေးပို့လိုက်သော မည်သည့်အမိန့်ကိုမဆို မျက်စိမှိတ် မယုံကြည်ရချေ။ ၎င်းသည် သတ်မှတ်ထားသော ဘောင်အတွင်း၌ ရှိမရှိ၊ ပေးအပ်ထားသော အာဏာ (Permissions) ထက် ကျော်လွန်နေခြင်း ရှိမရှိကို အမြဲတမ်း စစ်ဆေးရမည်။ ဤသည်မှာ "စာရေးတော်" ကို ခိုင်းစေရာတွင် ဘေးနားမှ "ကြီးကြပ်ရေးမှူး" တစ်ယောက် ထားရှိခြင်းနှင့် တူပေသည်။

### နိဂုံး - လက်ရှိ အခြေအနေနှင့် အနာဂတ်

ယနေ့ခေတ်တွင် OpenAI, Anthropic နှင့် Google ကဲ့သို့သော ကုမ္ပဏီကြီးများသည် ၎င်းတို့၏ Model များတွင် Tool Calling ကို အကောင်းဆုံး ဖြစ်အောင် လေ့ကျင့်ပေးထားကြသည်။ သို့သော် ၎င်းသည် ပြီးပြည့်စုံသည်ဟု မဆိုနိုင်သေးချေ။ Agent များသည် Tool များကို မှားယွင်းစွာ သုံးစွဲခြင်း၊ အဆင့်များကို ကျော်သွားခြင်း သို့မဟုတ် အဆုံးမရှိသော Loop ထဲသို့ ရောက်သွားခြင်းများ ရှိနေဆဲ ဖြစ်သည်။

အင်ဂျင်နီယာတစ်ယောက်အနေဖြင့် သင်၏ တာဝန်မှာ ထိုအမှားများကို ကြိုတင် ခန့်မှန်းနိုင်ရန်နှင့်၊ Agent က အမှားလုပ်ခဲ့လျှင်ပင် စနစ်တစ်ခုလုံး မပြိုလဲသွားစေရန် တည်ဆောက်ဖို့ ဖြစ်သည်။ Tool Calling သည် Agentic AI ၏ "လက်" များ ဖြစ်သကဲ့သို့၊ ထိုလက်များကို ထိန်းချုပ်သော "ကြိုး" များမှာမူ သင်၏ ကုဒ်များသာ ဖြစ်ပေသည်။

နောက်ပိုင်း tool calling တွင် **Programmatic Tool Calling (PTC)** ကဲ့သို့သော pattern များလည်းပေါ်လာသည်။ အလွယ်ပြောရလျှင် model သည် tool တစ်ခုချင်းစီကို JSON call အဖြစ်အပြန်အလှန်ခေါ်နေမည့်အစား code execution environment ထဲတွင် code ရေးပြီး tool workflow ကို run စေခြင်းဖြစ်သည်။ ဒီစာအုပ်ထဲတွင် ထိုသဘောကို Chapter 13 Travis-2 case study မှာ Playwright TypeScript ကို runtime boundary ထဲ run စေထားသောနေရာတွင် ပြန်မြင်နိုင်သည်။

နောက်အခန်းတွင်မူ ဤ Agent များသည် အလုပ်တစ်ခုကို မည်သို့ စဉ်းစားဆင်ခြင်ပြီး အဆင့်ဆင့် လုပ်ဆောင်သွားသနည်း ဆိုသည့် **Agent Loop** အကြောင်းကို ဆက်လက် ဆွေးနွေးသွားပါမည်။

**Source Notes &amp; Technical Facts for the Curious:**

- **JSON Schema:** A vocabulary that allows you to annotate and validate JSON documents, used extensively in Tool Definitions.
- **Zero-shot Tool Use:** The ability of a model to use a tool correctly based only on its description, without prior examples.
- **Few-shot Prompting:** Providing the model with a few examples of tool calls to improve its accuracy.
- **Parallel Tool Calling:** The ability of modern models to call multiple tools in a single response.
- **Programmatic Tool Calling:** Code execution environment ထဲတွင် model ရေးသော code က tools များကို programmatically ခေါ်ပြီး intermediate round-trip/context pressure ကိုလျှော့သော pattern ဖြစ်သည်။
- **Prompt Injection via Tools:** A security risk where the output of a tool (e.g., a web page content) contains instructions that hijack the agent's behavior.
