# 02-agent-vs-workflow-vs-automation

### အလိုအလျောက်စနစ်နှင့် ကိုယ်စားလှယ်ကြားရှိ ချောက်ကမ္ဘာ

ကျွန်ုပ်တို့သည် ယခင်အခန်းတွင် AI ဟု ခေါ်ဝေါ်နေကြသော အရာ၏ နောက်ကွယ်မှ "စာရေးတော်" (LLM) အကြောင်းကို ဆွေးနွေးခဲ့ကြသည်။ ယခုအခါ ထိုစာရေးတော်ကို အသုံးချ၍ ကျွန်ုပ်တို့ တည်ဆောက်မည့် စနစ်များ၏ အမျိုးအစားကွဲပြားပုံကို လေ့လာကြပါစို့။ နည်းပညာလောကတွင် "အလိုအလျောက်စနစ်" (Automation) နှင့် "ကိုယ်စားလှယ်စနစ်" (Agentic System) ကို ရောထွေးနေကြသူများ ဒုနဲ့ဒေး ရှိသည်။ ၎င်းတို့သည် ရထားလုံးနှင့် ပင်လယ်ကူးသင်္ဘောကို အတူတူပင်ဟု ထင်မှတ်နေကြသူများ ဖြစ်သည်။

ရထားလုံး (Traditional Automation) သည် သတ်မှတ်ထားသော သံလမ်းပေါ်တွင်သာ မောင်းနှင်ရသည်။ ၎င်းတွင် တိကျသော လမ်းကြောင်းရှိသည်။ "ဘူတာ A မှ ဘူတာ B သို့ သွားပါ" ဟု ခိုင်းလျှင် ၎င်းသည် ထိုလမ်းအတိုင်းသာ သွားမည်။ လမ်းတွင် သစ်ပင်လဲနေပါက ရထားသည် ထိုသစ်ပင်ကို တိုက်မိပြီး တိမ်းမှောက်သွားပေလိမ့်မည်။ အဘယ်ကြောင့်ဆိုသော် ၎င်းတွင် "ရှောင်ကွင်းရန်" ဟူသော အသိဉာဏ် မရှိသောကြောင့် ဖြစ်သည်။ ဤသည်မှာ ကျွန်ုပ်တို့ နှစ်ပေါင်းများစွာ သုံးစွဲလာခဲ့သော Deterministic Software များ၏ သဘာဝပင် ဖြစ်သည်။ ၎င်းတို့သည် ခန့်မှန်းရ လွယ်ကူသော်လည်း၊ ပြောင်းလဲနေသော ပတ်ဝန်းကျင်တွင်မူ အသုံးမဝင်လှချေ။

သို့သော် ပင်လယ်ကူးသင်္ဘော (Agentic System) မှာ မူ ထိုသို့မဟုတ်။ သင်္ဘောကပ္ပတိန်အား "ရန်ကုန်မှ စင်ကာပူသို့ သွားပါ" ဟု ခိုင်းလိုက်လျှင် သူသည် သံလမ်းအတိုင်း မောင်းနှင်ရန် မလိုအပ်ချေ။ လမ်းတွင် မုန်တိုင်းတွေ့လျှင် ရှောင်မည်၊ ကျောက်ဆောင်ရှိလျှင် ကွေ့မည်။ သူ၏ ပန်းတိုင် (Goal) မှာ စင်ကာပူသို့ ရောက်ရှိရန် ဖြစ်ပြီး၊ ထိုသို့ ရောက်ရှိရန်အတွက် လိုအပ်သော ဆုံးဖြတ်ချက်များကို သူကိုယ်တိုင် ချမှတ်သွားခြင်း ဖြစ်သည်။ ဤသည်မှာ Agentic AI ၏ အနှစ်သာရပင် ဖြစ်သည်။ ကျွန်ုပ်တို့သည် "မည်သို့လုပ်ရမည်" (How) ကို မခိုင်းတော့ဘဲ "ဘာဖြစ်ချင်သည်" (What) ကိုသာ ခိုင်းစေတော့ခြင်း ဖြစ်သည်။

### စက်ရုပ်၏ ဦးနှောက်နှင့် အလုပ်လုပ်ပုံ အဆင့်ဆင့်

Agent တစ်ခုကို တည်ဆောက်ရာတွင် ကျွန်ုပ်တို့သည် LLM ကို "ဦးနှောက်" အဖြစ် အသုံးပြုကြသည်။ သို့သော် ဦးနှောက်ရှိရုံဖြင့် အလုပ်မဖြစ်ချေ။ ထိုဦးနှောက်ကို မည်သို့ အသုံးချမည်နည်းဟူသော "လုပ်ငန်းစဉ်" (Workflow) လိုအပ်သည်။ ဤနေရာတွင် ကျွန်ုပ်တို့သည် အဆင့်သုံးဆင့်ကို သတိပြုရမည် -

၁။ **Linear Workflow (မျဉ်းဖြောင့် လုပ်ငန်းစဉ်):** ဤသည်မှာ အဆင့် (၁) ပြီးလျှင် (၂)၊ (၂) ပြီးလျှင် (၃) သွားသော စနစ်ဖြစ်သည်။ ဥပမာ - အီးမေးလ်ကို ဖတ်ပါ၊ ဘာသာပြန်ပါ၊ ပြီးလျှင် သိမ်းဆည်းပါ။ ဤနေရာတွင် LLM သည် အဆင့်တစ်ခုချင်းစီတွင် "ဘာသာပြန်သူ" အဖြစ်သာ ပါဝင်သည်။ ၎င်းတွင် ကိုယ်ပိုင် ဆုံးဖြတ်ချက် ချပိုင်ခွင့် မရှိသလောက်ပင်။ ဤစနစ်သည် အလွန် ဘေးကင်းသော်လည်း၊ ရှုပ်ထွေးသော ပြဿနာများကို မဖြေရှင်းနိုင်ချေ။

၂။ **Conditional Workflow (အခြေအနေပြ လုပ်ငန်းစဉ်):** ဤစနစ်တွင်မူ "အကယ်၍ အီးမေးလ်သည် အရေးကြီးပါက အကြောင်းပြန်ပါ၊ အရေးမကြီးပါက ဖျက်ပစ်ပါ" ဟူသော ဆုံးဖြတ်ချက်များ ပါဝင်လာသည်။ ဤနေရာတွင် LLM သည် "အရေးကြီးသလား၊ မကြီးဘူးလား" ဆိုသည်ကို ဆုံးဖြတ်ပေးရသူ ဖြစ်လာသည်။ ၎င်းသည် ရထားလမ်းပေါ်တွင် လမ်းခွဲ (Switch) များ ရှိနေခြင်းနှင့် တူသည်။

၃။ **Agentic Loop (ကိုယ်စားလှယ် ကွင်းဆက်):** ဤသည်မှာ အဆင့်မြင့်ဆုံး ဖြစ်သည်။ Agent သည် မိမိ၏ အလုပ်ကို မိမိဘာသာ ပြန်လည် စစ်ဆေးသည်။ "ငါ လုပ်လိုက်တာ မှန်ရဲ့လား"၊ "ပန်းတိုင်ကို ရောက်ပြီလား" ဟု မေးခွန်းထုတ်သည်။ အကယ်၍ မရောက်သေးပါက အစမှ ပြန်လည် ကြိုးစားသည်။ ဤသည်မှာ "ကိုယ်ပိုင်အသိ" (Autonomy) ရှိသော စနစ်ဖြစ်ပြီး၊ ကျွန်ုပ်တို့ ဤစာအုပ်တွင် အဓိကထား ဆွေးနွေးမည့် အကြောင်းအရာလည်း ဖြစ်သည်။

### ကပ္ပတိန်သည် အရက်မူးနေပါသလား

Agentic System တစ်ခုကို တည်ဆောက်ရာတွင် အင်ဂျင်နီယာများ ကြုံတွေ့ရသော အကြီးမားဆုံး ပြဿနာမှာ "ယုံကြည်မှု" ဖြစ်သည်။ ကျွန်ုပ်တို့၏ "သင်္ဘောကပ္ပတိန်" (LLM) သည် တစ်ခါတစ်ရံတွင် အရက်မူးနေသော ကပ္ပတိန်နှင့် တူတတ်သည်။ သူသည် အလွန်ပင် စကားတတ်သော်လည်း၊ လက်တွေ့တွင်မူ လမ်းမှားသို့ မောင်းနှင်သွားတတ်သည်။ ၎င်းကို Software Engineering တွင် "Non-deterministic" ဟု ခေါ်သည်။ ဆိုလိုသည်မှာ တူညီသော မေးခွန်းကို မေးသော်လည်း အဖြေမှာ တစ်ခါနှင့်တစ်ခါ မတူဘဲ ပြောင်းလဲနေခြင်း ဖြစ်သည်။

ရိုးရိုး Software များတွင်မူ `2 + 2` သည် အမြဲတမ်း `4` သာ ဖြစ်သည်။ သို့သော် LLM ကို `2 + 2` မေးလျှင် ၎င်းက `4` ဟု ဖြေကောင်း ဖြေလိမ့်မည်၊ သို့မဟုတ် "သင်္ချာပညာရှင်ကြီး ပိုက်သာဂိုးရပ်စ်၏ အဆိုအရ ၄ ဖြစ်ပါသည်" ဟု ရှည်ရှည်ဝေးဝေး ဖြေကောင်း ဖြေလိမ့်မည်။ ဤသို့ ခန့်မှန်းရ ခက်ခဲခြင်းသည် အင်ဂျင်နီယာများအတွက် ငရဲခန်းပင် ဖြစ်သည်။ ကျွန်ုပ်တို့သည် ခိုင်မာသော (Robust) စနစ်များကို တည်ဆောက်လိုသော်လည်း၊ ကျွန်ုပ်တို့၏ အခြေခံ အုတ်မြစ်ဖြစ်သော LLM မှာမူ သဲပေါ်တွင် ဆောက်ထားသော အိမ်ကဲ့သို့ ယိမ်းယိုင်နေတတ်သည်။

ထို့ကြောင့် ကျွန်ုပ်တို့သည် "အရက်မူးနေသော ကပ္ပတိန်" ကို ထိန်းကျောင်းရန်အတွက် "သင်္ဘောသားများ" နှင့် "မြေပုံများ" ကို စနစ်တကျ ပြင်ဆင်ပေးရမည်။ ဤသည်ကို **Orchestration** ဟု ခေါ်သည်။ Agent သည် မိမိဘာသာ ဆုံးဖြတ်ချက် ချနိုင်သော်လည်း၊ ထိုဆုံးဖြတ်ချက်များကို ကျွန်ုပ်တို့ သတ်မှတ်ထားသော "ဘောင်" (Constraints) အတွင်း၌သာ ရှိစေရမည်။ ဥပမာအားဖြင့် - "မင်း ကြိုက်သလို ဆုံးဖြတ်နိုင်တယ်၊ ဒါပေမဲ့ ပိုက်ဆံ တစ်သောင်းထက် ပိုသုံးမယ်ဆိုရင် ငါ့ကို အရင်မေးပါ" ဟူသော ကန့်သတ်ချက်မျိုး ဖြစ်သည်။

### အလုပ်သမားနှင့် ကိရိယာတန်ဆာပလာများ

Agent တစ်ခုသည် ၎င်း၏ ဦးနှောက် (LLM) သက်သက်ဖြင့် ဘာမှ မလုပ်နိုင်ချေ။ ၎င်းတွင် "လက်" များ လိုအပ်သည်။ ထို လက် များကို ကျွန်ုပ်တို့က **Tools** သို့မဟုတ် **Functions** ဟု ခေါ်ကြသည်။ Agent သည် အလုပ်တစ်ခုကို လုပ်ရန်အတွက် မည်သည့် Tool ကို သုံးရမည်ကို သိရမည်။ ဥပမာ - "မနက်ဖြန် ရာသီဥတု ဘယ်လိုရှိမလဲ" ဟု မေးလျှင် Agent သည် ၎င်း၏ မှတ်ဉာဏ်ထဲတွင် ရှာမနေဘဲ "Weather API" ဟူသော Tool ကို ခေါ်ယူ အသုံးပြုရမည် ဖြစ်သည်။

ဤနေရာတွင် အရေးကြီးသော အချက်မှာ Agent သည် Tool ကို "ခေါ်ယူခြင်း" (Calling) သာ လုပ်နိုင်ပြီး၊ Tool ကို "အကောင်အထည်ဖော်ခြင်း" (Execution) ကိုမူ ကျွန်ုပ်တို့၏ စနစ် (Harness) က လုပ်ဆောင်ပေးရခြင်း ဖြစ်သည်။ ဤသည်မှာ အာဏာခွဲဝေမှုပင် ဖြစ်သည်။ Agent ကို အရာရာ လုပ်ခွင့်ပေးထားခြင်းသည် အန္တရာယ် ရှိသည်။ ကျွန်ုပ်တို့သည် Agent အား "ဓား" ကို ပေးထားသော်လည်း၊ ထိုဓားဖြင့် ဘာကို လှီးရမည်ဆိုသည်ကိုမူ ကျွန်ုပ်တို့က စောင့်ကြည့်နေရမည် ဖြစ်သည်။

Agentic AI ၏ အနာဂတ်သည် ဤ Tool များကို မည်မျှ ကျွမ်းကျင်စွာ အသုံးချနိုင်သနည်း ဆိုသည့်အပေါ်တွင် မူတည်နေသည်။ ယခုအခါ **MCP (Model Context Protocol)** ကဲ့သို့သော နည်းပညာများ ပေါ်ပေါက်လာပြီး၊ Agent များသည် မတူညီသော Software များ၊ Database များနှင့် လွယ်ကူစွာ ဆက်သွယ်နိုင်လာကြပြီ ဖြစ်သည်။ ဤသည်မှာ ကျွန်ုပ်တို့၏ "စာရေးတော်" ကို ကမ္ဘာ့အဆင့်မီ "စီမံခန့်ခွဲသူ" တစ်ဦး ဖြစ်လာစေရန် လေ့ကျင့်ပေးနေခြင်းပင် ဖြစ်သည်။

### နိဂုံး - ရွေးချယ်ရမည့် လမ်းကြောင်း

သင်သည် Software တစ်ခုကို တည်ဆောက်တော့မည်ဆိုပါက မိမိကိုယ်ကို ပြန်မေးပါ။ "ငါ့မှာ တိကျတဲ့ လမ်းကြောင်း ရှိသလား"။ အကယ်၍ ရှိပါက ရိုးရိုး Automation (Workflow) ကိုသာ သုံးပါ။ ၎င်းသည် မြန်ဆန်သည်၊ ဈေးသက်သာသည်၊ ယုံကြည်ရသည်။ သို့သော် သင်၏ ပြဿနာသည် ရှုပ်ထွေးနေပြီး၊ အခြေအနေအပေါ် မူတည်၍ ပြောင်းလဲ ဆုံးဖြတ်ရမည်ဆိုပါက Agentic System ကို ရွေးချယ်ပါ။

သို့သော် သတိပြုရန်မှာ Agentic System သည် "မှော်တုတ်တံ" မဟုတ်ချေ။ ၎င်းသည် ပိုမို ရှုပ်ထွေးသည်၊ ကုန်ကျစရိတ် ပိုများသည်။ အရေးကြီးဆုံးမှာ ၎င်းကို ထိန်းကျောင်းရန်အတွက် ပိုမို မြင့်မားသော အင်ဂျင်နီယာ အတတ်ပညာ လိုအပ်ခြင်းပင် ဖြစ်သည်။ ကျွန်ုပ်တို့သည် စက်ပစ္စည်းများကို အာဏာအပ်နှင်းနေခြင်း ဖြစ်သောကြောင့်၊ ထိုအာဏာသည် ကျွန်ုပ်တို့ကို ပြန်လည် အန္တရာယ် မပေးနိုင်စေရန် အမြဲတမ်း သတိချပ်နေရမည် ဖြစ်သည်။

နောက်အခန်းတွင်မူ ဤ Agent များ၏ အသက်သွေးကြောဖြစ်သော **Tool Calling** နှင့် ၎င်းတို့၏ "လက်"  များ မည်သို့ အလုပ်လုပ်သည်ကို အသေးစိတ် ဆက်လက် လေ့လာသွားပါမည်။

**Source Notes &amp; Technical Facts for the Curious:**

- **Deterministic vs. Non-deterministic:** Deterministic systems always produce the same output for a given input, while non-deterministic systems (like LLMs) can vary.
- **Autonomy Levels:** From simple scripts to fully autonomous agents that can set their own sub-goals.
- **Orchestration Frameworks:** Tools like LangChain, CrewAI, and AutoGen that help manage agent workflows.
- **Model Context Protocol (MCP):** An open standard that enables developers to build secure, two-way connections between their data sources and AI models.
- **Human-in-the-loop (HITL):** A design pattern where a human must approve or intervene in an agent's decision-making process.
